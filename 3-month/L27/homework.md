## Questions

1. The Single Responsibility Principle says a class should have "only one reason to change." Explain what "reason to change" actually means — it is not about the number of methods. Give a real example of a class from a backend system (like a user service, an order processor, or a payment handler) that violates SRP, and describe how you would split it and why.

2. The Open/Closed Principle is often misunderstood. Explain it in your own words, then describe a real scenario where you added a new feature to a system and had to modify existing, already-tested code. What problems did that cause? How would following OCP have prevented those problems?

3. The Dependency Inversion Principle says to "depend on abstractions, not concretions." Explain what this means for testability — how does injecting an abstract `Logger` or `EmailService` instead of instantiating it inside a class make unit testing easier? Give a concrete example comparing the two approaches.

4. DRY (Don't Repeat Yourself) goes beyond just avoiding copy-paste. Explain the deeper meaning of DRY — why is duplicated *knowledge* (business rules, validation logic, calculations) more dangerous than duplicated *code*? Describe a real bug that could result from violating DRY in a financial or medical system.
