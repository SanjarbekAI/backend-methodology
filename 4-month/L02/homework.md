## Questions

1. Explain the purpose of a **branch** in Git using a real-world analogy. Then describe the professional rule: why should the `main` branch always contain only working, tested code? What could go wrong for a team if everyone commits their experimental work directly to `main`?

2. A **merge conflict** happens when two developers change the same line in the same file. Describe the step-by-step process for resolving a merge conflict in Git. Why does Git not just automatically pick one version — what would be the danger of that?

3. Compare `git fetch` and `git pull`. What does each command do, and when would a developer choose `fetch` over `pull`? Describe a scenario where using `git pull` at the wrong moment could cause problems for a developer working in a team.

4. A teammate says: "I just work alone, so I don't need branches — I'll just commit everything to `main`." Write a convincing response explaining at least two concrete situations where branches would protect even a solo developer from serious problems.
